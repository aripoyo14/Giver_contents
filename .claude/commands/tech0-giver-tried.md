# tech0-giver-tried

ありちゃんが「実際に試してみた」体験・感想をチャットで受け取り、LINE Giverコンテンツ1件を生成する。

## このコマンドの使い方

ありちゃんがチャットで以下を話しかけるだけでOK:
- 試したツール・機能の名前（または記事URL）
- 実際にやってみたこと
- 良かった点・惜しかった点
- あれば「もっとこうなればいいのに」という一言

例:
> 「Notionの議事録自動生成を試してみた。会議中に録音しておくと終わったあとに要約してくれる。日本語の精度が思ったより良かった。ただ専門用語は聞き取り間違える。Qiitaで紹介されてた。」

## 実行手順

### Step 0: ユーザーストーリーを確認する

`docs/user-stories.md` を読み、ペルソナ（関心はあるが「何をすればいいか分からない」状態）を頭に入れる。

### Step 1: ありちゃんの入力を整理する

チャット内容から以下を読み取る:

| 項目 | 内容 |
|------|------|
| トピック | 何のツール・機能・話題か |
| 元記事 | URL またはタイトル（ない場合は「ありちゃん体験談」として扱う） |
| やってみたこと | 実際に試した操作・手順 |
| 良かった点 | うまくいったこと・驚いたこと |
| 惜しかった点 | 不満・限界・改善してほしい点 |
| 発表のみか | 実際に使えるか、発表・予告だけか |

### Step 2: LINE文面を生成する

**フォーマット（厳守）:**
```
【今日のテック/AIニュース】

📌 [トピック名：15文字以内・キャッチー]

[何が起きたか：2〜3行、平易な言葉・技術用語なし]

💬 実際に試してみた——[ありちゃんの体験・感想・惜しいポイント。1〜2行。一人称・話し言葉寄りに]

[自分ごと化フレーズ：「〜な人——こうすると〇〇できます。」形式]
```

**「💬 実際に試してみた——」の書き方ルール:**
- ありちゃんの言葉をもとに、一人称（「私は」「試してみたら」）で整える
- 良かった点だけでなく、惜しかった点・正直な感想も含める（完璧に見せない）
- 「すごかった」「便利だった」の一言で終わらず、具体的な場面・数値・比較を入れる
- 文末は「〜でした」「〜でしたよ」など話しかけるトーンで

**事実確認ルール:**
- 「発表のみ」で実際に使えない場合は `（現在準備中、まもなく提供予定）` を [何が起きたか] に明記
- ありちゃんが実際に試していない機能については「試してみた」と書かない

**自分ごと化フレーズのルール:**
- 「〜な人——[具体的な操作]で[具体的な結果]。」の形
- 属性を1つ絞る（「〜している人」「〜がちな人」）
- 「仕事の効率が変わります」「差がつきます」のような抽象表現は禁止

**トーン:**
- 煽り・脅し表現は禁止（「波に乗るか流されるか」等）
- 読後感は「へぇ、面白い」「ちょっと試してみようかな」

### Step 3: drafts/ に保存する

`drafts/YYYY-MM-DD.md` に追記する（同日既存ファイルがあれば末尾に追記、なければ新規作成）。

```markdown
# LINE Giverコンテンツ案 YYYY-MM-DD

> 生成日時: YYYY-MM-DD HH:MM
> 種別: ありちゃん体験談

---

## 案（カテゴリ: すぐ試せる｜元記事: [タイトルまたはURL]）

[LINE文面]

---
```

### Step 4: GitHubにコミットしてPRを作成する

```bash
git add drafts/YYYY-MM-DD.md
git commit -m "draft: YYYY-MM-DD ありちゃん体験談コンテンツ案"
gh pr create \
  --title "[DRAFT] YYYY-MM-DD Giver LINE ありちゃん体験談" \
  --body "## 概要
ありちゃんが実際に試したツール・機能をもとに生成したコンテンツ案です。

## レビュー依頼
- [ ] 「💬 実際に試してみた」の文面・トーンの確認
- [ ] 事実誤認がないか（発表のみのものを「使える」と書いていないか）
- [ ] 自分ごと化フレーズの具体性確認
- [ ] 承認後、LINEへ手動配信"
```
