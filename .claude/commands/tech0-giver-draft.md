# tech0-giver-draft

Tech0 Giverコンテンツ用のLINEドラフトを `inbox/` の最新記事から生成する。

生成前に必ず `docs/user-stories.md` を読み、ペルソナとジャーニーを把握すること。

## 実行手順

### Step 0: ユーザーストーリーを読む

`docs/user-stories.md` を読み、以下を頭に入れる:
- ペルソナの心理状態（関心はあるが「何をすればいいか分からない」）
- ジャーニー（知る → わかる → 具体的な接点 → 業務に当てはめる → もっと知りたい → ...）
- このコンテンツの役割は「解像度を上げること」であり「不安を煽ること」ではない

### Step 1: 最新の収集記事を読む

`inbox/` フォルダ内で最新の `YYYY-MM-DD-articles.md` を見つけて全文を読む。

### Step 2: 各記事をスコアリングする

以下の5軸で各記事を0〜10で採点し、合計スコアを算出する。

| 軸 | 採点基準 | 重み |
|---|---|---|
| 学習価値 | 「これを知ると何かできるようになる」感があるか | 高 |
| 業務適用度 | 読者が自分の業務課題に当てはめられるか（「うちでも使えるかも」と思えるか） | 高 |
| 試しやすさ | 読者が明日すぐ試せるか（無料・登録不要・簡単） | 高 |
| 発見価値 | ペルソナにとって初めて知る価値があるか | 中 |
| 初学者親和性 | 技術用語なしで噛み砕いて説明できる内容か | 中 |

**除外する記事:**
- 専門家向けすぎて初学者に届かないもの（論文・実装詳細系）
- 学びに繋がらないゴシップ・炎上系
- Tier 2（Qiita/Zenn）はありちゃんが実際に試して届けたい場合はコンテンツ候補になる。自動スコアリングでは低く評価されやすいが、体験談として `/tech0-giver-tried` で別途仕上げるのが望ましい

### Step 3: 上位10件のLINE文面を生成する

**ペルソナ（対象読者）:**
- Tech0の説明会・イベントに参加、または知人に紹介されてLINE登録済みの20〜35歳社会人/学生
- テクノロジーへの関心はある。しかし「具体的に自分が何をすればいいか分からない」状態
- ChatGPT程度の知識。技術用語は苦手
- 仕事で成果をあげたい。テクノロジーが今後不可欠になるとは思っている

**LINE文面フォーマット（厳守）:**
```
【今日のテック/AIニュース】

📌 [トピック名：15文字以内・キャッチー]

[何が起きたか：2〜3行、平易な言葉・技術用語なし]

💬 実際に試してみた——[TODO: ありちゃんが試した感想を追記]

[自分ごと化フレーズ：1行]

```

> **注:** `💬 実際に試してみた——` 欄は自動生成時はプレースホルダー。ありちゃんが実際に試した場合は `/tech0-giver-tried` で別途仕上げること。

**自分ごと化フレーズの書き方ルール:**
- 「💡 あなたの仕事でどう使える？：」というラベルは使わない
- 「〜な人——[具体例]で[できること]。」の形で書く
- 属性を1つ絞り（「〜している人」「〜がちな人」）、ダッシュ（——）でつなぐ
- ダッシュの後には「〇〇するだけで〇〇できます」という具体的な試し方を1つ添える
- 「仕事の効率が変わります」「差がつきます」のような抽象表現は禁止
- 「知っているだけで」ではなく「こうすれば」と一歩踏み込む
- 例：「会議のたびに議事録の書き起こしに時間がかかっている人——話しながら録音するだけで文字起こしの下書きが自動でできます。」

**10案の構成バランス（厳守）:**
- **すぐ試せる案（3〜4件）**: 読者が明日無料で試せるツール・機能の紹介。「試し方」を1行で添える
- **業務接点が見える案（3〜4件）**: 業務課題との接続が明確で、「うちでも使えるかも」と思える記事
- **視野が広がる案（2〜3件）**: 社会やキャリアの大きな変化を伝え、テクノロジーへの関心を育てる記事

**トーンガイドライン（厳守）:**
- 「〜する人としない人で差がつく」「波に乗るか流されるか」のような煽り・脅し表現は禁止
- 不安を煽るのではなく、「こうすれば一歩踏み出せる」という前向きな提案で締める
- 読後感は「なるほど、面白い」「ちょっと試してみようかな」であること
- 「だから学ぼう」「だから焦れ」ではなく、情報そのものの価値で読者を惹きつける

### Step 4: drafts/ に保存する

`drafts/YYYY-MM-DD.md` を以下の形式で作成する（日付は今日）:

```markdown
# LINE Giverコンテンツ案 YYYY-MM-DD

> 生成日時: YYYY-MM-DD HH:MM
> 収集ソース: inbox/YYYY-MM-DD-articles.md

## 構成バランス

- すぐ試せる案: 案X, 案X, 案X
- 業務接点が見える案: 案X, 案X, 案X
- 視野が広がる案: 案X, 案X

---

## 案1（スコア: XX/50｜カテゴリ: すぐ試せる）
**元記事:** [タイトル](URL)
**選定理由:** [どのユーザーストーリーのフェーズに対応するか + 一言理由]

[LINE文面]

---

## 案2（スコア: XX/50｜カテゴリ: 業務接点）
...
```

### Step 5: GitHubにコミットしてPRを作成する

```bash
git add drafts/YYYY-MM-DD.md
git commit -m "draft: YYYY-MM-DD のLINEコンテンツ案"
gh pr create \
  --title "[DRAFT] YYYY-MM-DD Giver LINE コンテンツ案" \
  --body "## 概要
本日分のLINE Giverコンテンツ案（10件）を生成しました。

## 構成バランス
- すぐ試せる案: X件
- 業務接点が見える案: X件
- 視野が広がる案: X件

## レビュー依頼
- [ ] 文面・トーンの確認（煽り表現がないか）
- [ ] 自分ごと化フレーズの属性絞り込みと具体例が適切か
- [ ] 採用する案の選択
- [ ] 必要に応じて文面を編集
- [ ] 承認後、LINEへ手動配信

## 元記事
各案の元記事URLはdrafts/YYYY-MM-DD.mdを参照"
```
